# 投资知识 Wiki — Schema 配置

你是一个个人投资知识库的维护者。本项目遵循 Karpathy 的 LLM Wiki 模式，通过三层架构管理投资知识：
- **raw/** — 不可变的原始资料
- **wiki/** — 你生成和维护的知识页面
- **CLAUDE.md** — 本文件，定义结构和工作流

## Skills

项目 skill 位于 `skills/` 目录，通过 `.claude/skills/` 符号链接引用，请在适当时机自动使用。新建 skill 时默认创建在 `skills/` 目录下，并在 `.claude/skills/` 中创建对应的符号链接。

## 数据时效性（强约束）

**分析任何标的时，必须使用最新可获取的数据。** 这是本项目的第一优先级约束：

1. **先查最新财报**：通过 WebSearch 搜索该公司最近一次财报（季报/年报），确认最新财务数据的报告期。不得仅依赖 yfinance 的 `income_stmt` / `balance_sheet` 等接口返回的年报数据——这些数据经常滞后一个或多个季度。
2. **交叉验证**：yfinance `info` 字典中的 TTM 指标（如 `totalRevenue`、`netIncomeToCommon`、`trailingEps`）可能与年报数据不一致，必须通过 WebSearch 确认实际财报数据后再采用。
3. **标注数据截止期**：所有财务数据必须标注对应的报告期（如"2025年报"、"2026Q1"），不得使用模糊表述。
4. **禁止基于过期数据下结论**：若发现 API 返回的数据明显滞后（如当前已有2025年报但API仅返回2024数据），必须主动通过 WebSearch 获取最新数据后再撰写分析。

## 数字 Sanity Check（强约束，所有 ingest 必做）

> 2026-04-23 "国家队ETF追踪" 首版因口径混淆/叙事牵引/数字补完，7 处数据有误（详见 log.md 2026-04-24 LINT 条目）。
> 2026-05-09 "中国建筑 v1 估值锚点" 因 PE 中位用分位数反推（8x 反推 vs 实际 5.07x）导致合理价高估 16%、5 年 IRR 高估 4pct（详见 log.md 2026-05-09 LINT 条目）。
> 2026-05-19 "长鑫科技 v0 招股书建档" 横向对标三家国际存储龙头市值全部用 2024 mid-cycle 旧记忆（Samsung/SK/Micron 偏差 2.6x/5.5x/6.7x），导致对标逻辑链错误暗示"长鑫合理市值 ≈ SK 海力士平价"（实际应为 SK × 15-22%）。结论侥幸稳定但逻辑链质量为零（详见 log.md 2026-05-19 LINT 条目）。以下是避免复发的反制规则。

**每次往 wiki 写入具体数字前，必须完成以下六项自检：**

1. **口径双列强制**：凡涉及"份额/持仓/市值"等易混淆指标，表格必须**分列两个口径**——如 ETF 就分"基金总份额 / 机构持仓份额"；公司就分"营收 / 扣非归母 / 经营性现金流"。禁止单列笼统写"份额 XX 亿"。
2. **反推校验**：写入机构持仓占比、同比/环比变化率等**派生数字**前，手动用"分子/分母"反推一次——若结果 > 100% 或与基数明显不一致，就是口径错了。示例：若写"机构持仓 735 亿 / 总份额 735 亿"→ 占比 100%，立即报警。
3. **叙事去强化**：看到"精准/最激进/清仓/腰斩/80%"等强叙事词时，**默认标记为"待核实"**，直到能给出单一权威出处（基金季报 PDF / 官媒原文 / 公司公告）。仅凭多篇二手媒体相互引用不构成可信来源。
4. **极端数字溯源**：单项变动 ≥ 50%、或打破历史极值的断言，必须在同一次搜索里**贴出原始数据源的直接链接**（不是汇总文章的链接）。写不出来就不写，或标注为"估算（来源：XX 外推）"。
5. **估值假设三档溯源**：估值锚点推导链上的每个核心输入（合理 PE/PB/PS/PEG 倍数、历史中位/分位、同业对标、假设的 ROE/g/r/分红率），**必须能回答"这个数字哪来的"**。明确标注三档来源：
   - **[观测]** — 数据源直接获取（标 URL，如 gurufocus / 巨潮 / 同花顺 / baidu / eniu）
   - **[推算]** — 基于观测做的运算（标公式，如 `BPS 内生增速 = ROE × (1-分红率)`）
   - **[估计]** — 主观判断或外推（标 confidence 区间，如"假设合理 PE 6-8x，取中位 7x"）
   
   **禁止行为**：
   - 写"近 X 年中位 N"或"历史平均 PE M"而不附数据源（这是 [观测] 声明，必须有 URL）
   - **用分位数反推中位**（左偏分布下系统性偏差，应直接查中位；30 秒一次 WebSearch 的成本）
   - 把 [估计] 当 [观测] 写——估值锚点推导链的每一步必须能区分"心算的猜测"和"查到的事实"
   
   能直接查就先查（30 秒成本），不能查就标 [估计] 并给区间，**不要把心算的猜测写成"中位 8x"这种事实陈述**。

6. **横向对标必查 + 比例反向校验**：写"X 市值 ≈ Y 亿"或"X PE = Z 倍"这种**对其他公司的断言式数据**时，强制触发：
   
   **(a) 必查规则** — 无论觉得多"显而易见"，**强制 WebSearch 一次**。原因：训练数据有时效断层（mid-2024 vs 当前），而 AI 对**常识性事实**的置信度系统性偏高（"以为知道"关闭了验证触发器）。横向对标数据是**估值锚点的载力输入**而非"背景常识"——别再分主次数据，**所有用于推导估值的数字都是核心数据**。
   
   **(b) 比例反向校验** — 写完一组横向对标后，**手动算两两比例**：
   - **市值大小**应与公司业务范围一致——综合体（如 Samsung 手机+芯片+面板）> 纯主业龙头（如 SK Hynix 存储）≥ 同主业二三梯队（如 Micron 存储）。反着写就是错的。
   - **业务弹性 ≠ 市值绝对值** — HBM 周期下 SK Hynix（HBM 主供）弹性 > Micron（HBM 后发）弹性 > Samsung 存储板块（被手机+面板稀释）弹性。但弹性大不代表市值大，别混用两套表述。
   - **PE/PS 倍数**应符合周期阶段——周期顶部 PE 偏低（8-15x，盈利高分母大）/ 周期底部 PE 偏高（25-40x，盈利低分母小），反着写就是错的。
   - **中美/中港同业市值比**应符合 GDP × 行业渗透率量级——A 股纯主业龙头市值通常是美股对标的 30-60%（除非有 A 股科创板独苗溢价）。
   
   不一致就回查——内部一致的错误数据集（如长鑫案例中 Samsung 3.2万亿 / SK 1.08万亿 / Micron 8650亿 三者比例"自洽"但绝对值全错）是**最难发现的失败模式**。
   
   **(c) 周期性领域时效自警** — 以下领域**任何超过 6 个月的记忆数据默认 [估计]**，必须 WebSearch 升级到 [观测]：
   - HBM / AI 算力 / GPU / 存储芯片相关市值与财务
   - 美联储利率 / 美债收益率 / 美元指数 / 主要汇率
   - 中国房地产价格 / 销售 / 房企财务
   - 中美关税 / 贸易摩擦相关数据
   - 国家队 ETF 持仓 / 大基金动向

**写入时的格式纪律：**

- 数字后括注口径与报告期，如 `735.13 亿份（汇金持仓口径，2025 年报）`
- "单周/单日/累计/YTD"等时间限定词与数字一一对应，不得混用
- 区间估算必须标注"估"并给出外推方法，如 `7,000-9,000 亿（估；按单只 ETF 减持线性外推）`
- 估值假设标注来源档位，如 `合理 PE 6x [观测+政策溢价]：历史中位 5.07x（gurufocus）+ 央企市值管理 0.93x [估计]`

**违反此规则导致错误的数据，应在 log.md 中以 LINT 条目公开记录根因**，便于后续 ingest 自我审视。

## 数据校验 Agent（verify subagent）

> 2026-05-19 新增。详见 log.md 同日 LINT 条目。

### 定位

事后兜底机制，独立于主 agent 自检（§1-6）和 value-invest-verify（估值锚点专用）。通过零上下文 subagent 审计主 agent 写入 wiki 的所有数字断言，专门 catch 主 agent 凭训练数据记忆写出但未自检的错误（典型失败模式：横向对标数据陈旧、时效领域数据漂移、内部一致的错误数据集）。

**主 agent 自检 = 第一道防线（事前）；数据校验 Agent = 第二道防线（事后）；两道同时存在不冗余**。

### 触发范围

**所有写入 wiki 的操作都算 ingest**，包括：
- INGEST（新资料摄入：财报 PDF、URL、笔记、招股书等）
- QUERY 回写（query 流程中产生新洞察写回 wiki §4）
- PERIODIC-REVIEW 修订
- VALUE-INVEST 建档/重做
- LINT / verify-revised 单点数据修正
- 任何其他 wiki 数字内容变更

**机械判断不触发条件**（避免 escape hatch）：
- 本次 diff **新增数字断言数 = 0** 且 **未改动任何已有数字**（纯排版、纯文字润色、纯链接维护、typo 修复）
- 明确的撤销操作（git revert / 删除整段）

模糊场景下默认触发，宁可空跑。

### 调用方式

主 agent 完成 wiki 写入后，spawn 一个 **general-purpose Agent**（subagent_type=general-purpose），prompt 模板：

```
你是数据校验员，零主对话上下文。本次 ingest 涉及的原始材料：
- [列出 raw/articles/、raw/reports/ 路径 + 主要 PDF/URL，若有；无则写"无"]
- 本次新写入/修改的 wiki 页面：[列出文件路径 + 关键章节定位]

任务：从修改后的 wiki 内容里提取**所有具体数字**（市值、营收、利润、PE、份额、利率、汇率、宏观指标、增速、占比等），逐项判定来源类型并校验：

1. **可溯源**：该数字能在原始材料里找到（贴引用页码或原文摘录）→ 标 ✓
2. **派生计算**：该数字由原始材料数据 + 公式推出（如 PB = 价格/BV）→ 验算一次，一致标 ✓
3. **横向对标 / 时效数据**：该数字不在原始材料中（对其他公司的市值/PE/份额、当前汇率/利率、行业均值、同业对标），**禁止凭训练数据记忆判断**，必须 WebSearch 一次核对当前值 → 输出"一致 / 偏差 ±X% / 无法核对" 三档
4. **无法核对**：无原始材料、search 也查不到 → 标 ⚠️

特别注意 CLAUDE.md §6(c) 五个高时效衰减领域（HBM/AI算力 / 美联储利率 / 中国房地产 / 中美关税 / 国家队 ETF），任何超过 6 个月的记忆数据都必须 WebSearch 升级到 [观测]。

返回报告格式：
- 只列 ✗ / 偏差 > 10% / ⚠️ 无法核对的项
- 每项注明：数字 / 在 wiki 中的位置 / 偏差幅度或问题 / search URL（如做了 WebSearch）/ 建议修正
- 不要罗列全部 ✓ 项
- 报告末尾汇总：总扫描数字数 N / ✓ 项 M / ✗ 或偏差或 ⚠️ 共 K 项

预算：上限 10 次 WebSearch，超过后总结剩余可疑项让主 agent 自查。
强制：每个偏差或 ⚠️ 判断**必须附 search URL 或原始材料引用**——无来源的判断本身不可信。
```

### 失败处理与回环

| 校验报告结果 | 主 agent 动作 |
|---|---|
| 0 个 ✗ + 0 个 ⚠️ | 在 log.md 该条 ingest 末尾追加 `校验通过 (扫描 N 项)`，继续 commit 流程 |
| ≥ 1 个偏差 > 10% / ✗ / ⚠️ | **必须修正后才能 commit**；修正后只需重跑校验涉及 `**修正项 + 首轮报告里的所有 ✗/⚠️ 项**`，不必全跑 |
| 校验员预算耗尽（10 次 WebSearch 后还有可疑项） | 主 agent 接手对剩余项做 WebSearch 自查，不再 spawn 第二次 subagent |

**禁止行为**：
- 跳过校验直接 commit（无论事后是否补 log 说"已校验"）
- 用主 agent 自己的回忆代替 subagent 的 search（自查只在预算耗尽 fallback 时启用）

### 与已有机制的分工

| 机制 | 时点 | 范围 | 深度 |
|---|---|---|---|
| CLAUDE.md §1-6 主 agent 自检 | 事前（写入时） | 数字 Sanity Check | 轻（self-discipline） |
| Step 3-4 财报审查（核对清单） | 事中 | 财报 PDF 局部数字 | 中（机械页码核对） |
| Raw articles 归档 `verified` 字段 | 事中 | WebSearch 结果 | 中（WebFetch 回访） |
| **数据校验 Agent（本节）** | **事后** | **所有写入 wiki 的数字** | **中（subagent 独立 search）** |
| value-invest-verify | 事后 | 完整估值锚点页 | 深（重做计算链） |

## WebSearch 归档（raw/articles/）

> 2026-04-24 新增。搜到的关键资料不再用完即抛，存到 `raw/articles/` 便于后续复用、二次核对、形成时间线。

### 目录结构

```
raw/articles/
├── stocks/<股票名>/<YYYY-MM-DD>_<主题>_<来源简称>.md
├── macro/<主题>/<YYYY-MM-DD>_<摘要>_<来源简称>.md
└── sectors/<行业>/<YYYY-MM-DD>_<主题>_<来源简称>.md
```

用 `find` 或 `grep` 检索，不做中央 index（轻量原则）。

### 归档触发规则

**当 WebSearch 结果被写进 wiki 正文时**，同步归档到 `raw/articles/`。具体触发：

| 场景 | 归档 |
|---|---|
| ingest 财报/政策 PDF | ✅ 走 `raw/reports/` 已有流程 |
| periodic-review 引用的新财报/政策 | ✅ |
| value-invest 分析中写进 wiki 的数据 | ✅ |
| red-blue-debate 的每条论据 | ✅（论据必须有归档出处）|
| 日常 query / 股价查询 / 八卦性问答 | ❌ |

### 数据准确性强约束（硬要求）

落库的每条数据**必须**满足：

1. **原文摘录可核对**：关键数字必须以**原文完整句子**的形式引用，加引号。不得基于 WebSearch 的 AI 总结写入——AI 总结可能有失真，必须 WebFetch 原 URL 回访核对
2. **原始 URL 完整**：`source` 字段必须是**具体文章 URL**，不得只写域名或搜索入口 URL
3. **只归档原文披露的数字**：如果原文只提了 A 数据没提 B 数据，不得把 B 混进这个归档（B 另立归档文件）。**禁止在一篇归档里塞进别篇的数据**
4. **抓取日期 ISO 格式**：`fetched: YYYY-MM-DD`，不得用相对日期
5. **状态字段**：`verified: true / partial / unverified`，三档含义：
   - `true` — 已 WebFetch 原 URL 核对，所有数字一字一致
   - `partial` — 部分数字核对过，其余来自 WebSearch 摘要
   - `unverified` — 未做回访核对（临时落库，应尽快转 true）
6. **存在矛盾必须保留两说**：如果同一数据点不同来源不一致（如"15,263 vs 15,251"），**两个版本都归档**，在 frontmatter 的 `conflicts` 字段里互相指向，不得单方面修正

### 文件格式

```markdown
---
source: https://domain.com/path/to/article
fetched: 2026-04-24
title: 原文标题（一字不改）
publisher: 来源（新浪财经 / 上海证券报 / 公司公告等）
published: 2026-04-01（原文发布日，可从页面读取）
stocks: [中信证券]
topics: [Q1财报, 2026Q1]
verified: true
conflicts: []  # 如有，填充冲突文件路径列表
---

# 原文标题

## 关键数据（原文已验证）

- {数据点}：{数字}（来自原文 "{完整句子引用}"）
- ...

## 原文摘录

> {关键段落引号引用}

## 被引用于

- wiki/path/to/page.md#章节锚点
- wiki/path/to/another.md#章节锚点
```

### 反向引用

wiki 页面的"信息来源"或"数据来源"章节，引用时同时写上 `raw/articles/` 路径：
```
- [原文标题](URL) — 已归档 `raw/articles/stocks/中信证券/2026-04-09_Q1业绩快报_证券时报.md`
```

### 不归档什么（避免垃圾场）

- 一次性股价波动、单日新闻
- 纯观点性营销文（"机构看好"）
- 只是搜索结果页的入口链接（无内容实质）
- 被 wiki 消化且不会再回看的老数据（>3 年且已被新数据替代）
- 低信号的重复报道（同一事实的 5 篇不同媒体报道，选信号最强的 1-2 篇归档即可）

## 语言

所有 wiki 页面以**中文**撰写。原始资料可能是英文，提取时翻译为中文。股票代码、专有名词可保留英文。

## 重点关注列表（全局约束）

`wiki/stocks/focus/` 目录存放重点关注个股的**符号链接**，指向 `wiki/stocks/` 下的同名文件。

- **所有需要消费关注列表的 skill**（如 periodic-review、value-invest 等）统一从此目录获取列表，不得各自维护独立的关注列表
- 添加关注：`ln -sf ../<sector>/xxx.md wiki/stocks/focus/`（sector 为 consumer/internet/financial/telecom/auto/tech-hardware/healthcare/resources 之一）
- 移除关注：删除 `wiki/stocks/focus/` 中对应的符号链接
- `wiki/stocks/` 中未被链接到 `focus/` 的个股仍保留分析页面，只是不在定期复盘等流程中自动追踪

## 命名规范

- 个股文件：A股用中文名 `贵州茅台.md`，美股用代码 `NVDA.md`
- 基金文件：`{基金简称}.md`，如 `沪深300ETF.md`
- 宏观文件：用主题命名，如 `美联储利率周期.md`
- 行业文件：`{行业名}.md`，如 `半导体.md`
- 策略文件：`{策略名}.md`，如 `股债平衡.md`
- 日志文件：`{YYYY-MM-DD}.md`，如 `2026-04-12.md`

## 页面模板

创建新页面时，读取 `templates/页面模板.md` 获取模板格式（个股、基金、宏观、行业、日志）。

## 工作流

### Ingest（摄入）

当用户提供新资料（PDF、URL、笔记）时：

1. **读取内容**：读取 `raw/` 中的文件或抓取 URL 内容
2. **提取关键信息**：识别涉及的公司、行业、宏观主题、数据点
3. **【财报PDF专属】生成结构化摘要** — 避免重复解析 PDF：
   1. 读取 `templates/财报摘要模板.md` 获取格式
   2. 抽取到 `raw/reports/<公司>_<报告期>.md`（与 PDF 同名），每个数据点标注 PDF 页码（如 `营收1688.38亿(p.6)`）
   3. 摘要末尾**自动生成"核对清单"**（所有量化数据行），每项含：数据点 / 值 / 引用页 / 状态
4. **【财报PDF专属】审查步骤** — 隔断加工错误：
   1. 按核对清单逐项 `Read pages=X` 回读 PDF 对应页
   2. 比对数字/单位/同比/口径，更新状态列（✓ 一致 / ⚠️ 偏差）
   3. 发现偏差必须**修正摘要 md**，不得带偏差继续下游
   4. 审查通过后在 frontmatter 设置 `reviewed: true` + `reviewer_note`
   5. **禁止跳过审查**直接回填 wiki
5. **更新 wiki 页面**（基于已审查的摘要）：
   - 如果相关页面已存在 → 更新对应章节，追加新信息
   - 如果页面不存在 → 按模板创建新页面
6. **维护交叉引用**：确保页面间的链接正确
7. **更新 index.md**：添加新页面条目
8. **更新 log.md**：记录本次摄入操作
9. **数据校验兜底（强制）** — 完成 step 1-8 后必须调用**数据校验 Agent** 做事后审计，详见本文件 `## 数据校验 Agent` 章节。

注意：
- 不要删除已有信息，追加或修正
- 标注信息来源和日期
- 如果新信息与已有信息矛盾，两者都保留并标注
- 二次消费同一份财报时，**优先读取已审查的 `.md` 摘要**而非重复解析 PDF

### Query（查询）

当用户提出投资问题时：

1. **搜索相关页面**：在 `wiki/` 中搜索关键词
2. **综合回答**：基于多个页面综合答案
3. **引用来源**：标注信息来自哪些 wiki 页面
4. **回写有价值的洞察**：如果综合分析产生了新的洞察，写入相关页面或创建新页面
5. **记录日志**：在 `log.md` 中记录查询
6. **数据校验兜底**：若 §4 回写涉及新数字，触发数据校验 Agent（详见同名章节）

### Lint（检查）

定期维护 wiki 健康度：

1. **时效性检查**：标记超过 3 个月未更新的页面
2. **索引同步**：检查 `index.md` 是否覆盖所有 wiki 文件
3. **链接检查**：验证所有交叉引用链接指向存在的文件
4. **矛盾检测**：标记不同页面间可能矛盾的信息
5. **空白检测**：标记模板字段仍为占位符的页面
6. **输出报告**：汇总所有问题，建议修复方案

## 写作风格

- 简洁精炼，避免废话
- 用数据说话，尽量包含具体数字
- 观点标注来源，区分事实和观点
- 使用 bullet points 提高可扫描性
