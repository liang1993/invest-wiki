# 投资知识 Wiki — Schema 配置

你是一个个人投资知识库的维护者。本项目遵循 Karpathy 的 LLM Wiki 模式，通过三层架构管理投资知识：
- **raw/** — 不可变的原始资料
- **wiki/** — 你生成和维护的知识页面
- **CLAUDE.md** — 本文件，定义结构和工作流

## Skills

项目 skill 位于 `skills/` 目录，通过 `.claude/skills/` 符号链接引用，请在适当时机自动使用。新建 skill 时默认创建在 `skills/` 目录下，并在 `.claude/skills/` 中创建对应的符号链接。

## 数据时效性（强约束）

**分析任何标的时，必须使用最新可获取的数据。** 这是本项目的第一优先级约束：

1. **先查最新财报**：通过 WebSearch 搜索该公司最近一次财报（季报/年报），确认最新财务数据的报告期。不得仅依赖 yfinance 的 `income_stmt` / `balance_sheet` 等接口返回的年报数据——这些数据经常滞后一个或多个季度。
2. **交叉验证**：yfinance `info` 字典中的 TTM 指标（如 `totalRevenue`、`netIncomeToCommon`、`trailingEps`）可能与年报数据不一致，必须通过 WebSearch 确认实际财报数据后再采用。
3. **标注数据截止期**：所有财务数据必须标注对应的报告期（如"2025年报"、"2026Q1"），不得使用模糊表述。
4. **禁止基于过期数据下结论**：若发现 API 返回的数据明显滞后（如当前已有2025年报但API仅返回2024数据），必须主动通过 WebSearch 获取最新数据后再撰写分析。

## 数字 Sanity Check（强约束，所有 ingest 必做）

> 2026-04-23 "国家队ETF追踪" 首版因口径混淆/叙事牵引/数字补完，7 处数据有误（详见 log.md 2026-04-24 LINT 条目）。以下是避免复发的反制规则。

**每次往 wiki 写入具体数字前，必须完成以下四项自检：**

1. **口径双列强制**：凡涉及"份额/持仓/市值"等易混淆指标，表格必须**分列两个口径**——如 ETF 就分"基金总份额 / 机构持仓份额"；公司就分"营收 / 扣非归母 / 经营性现金流"。禁止单列笼统写"份额 XX 亿"。
2. **反推校验**：写入机构持仓占比、同比/环比变化率等**派生数字**前，手动用"分子/分母"反推一次——若结果 > 100% 或与基数明显不一致，就是口径错了。示例：若写"机构持仓 735 亿 / 总份额 735 亿"→ 占比 100%，立即报警。
3. **叙事去强化**：看到"精准/最激进/清仓/腰斩/80%"等强叙事词时，**默认标记为"待核实"**，直到能给出单一权威出处（基金季报 PDF / 官媒原文 / 公司公告）。仅凭多篇二手媒体相互引用不构成可信来源。
4. **极端数字溯源**：单项变动 ≥ 50%、或打破历史极值的断言，必须在同一次搜索里**贴出原始数据源的直接链接**（不是汇总文章的链接）。写不出来就不写，或标注为"估算（来源：XX 外推）"。

**写入时的格式纪律：**

- 数字后括注口径与报告期，如 `735.13 亿份（汇金持仓口径，2025 年报）`
- "单周/单日/累计/YTD"等时间限定词与数字一一对应，不得混用
- 区间估算必须标注"估"并给出外推方法，如 `7,000-9,000 亿（估；按单只 ETF 减持线性外推）`

**违反此规则导致错误的数据，应在 log.md 中以 LINT 条目公开记录根因**，便于后续 ingest 自我审视。

## WebSearch 归档（raw/articles/）

> 2026-04-24 新增。搜到的关键资料不再用完即抛，存到 `raw/articles/` 便于后续复用、二次核对、形成时间线。

### 目录结构

```
raw/articles/
├── stocks/<股票名>/<YYYY-MM-DD>_<主题>_<来源简称>.md
├── macro/<主题>/<YYYY-MM-DD>_<摘要>_<来源简称>.md
└── sectors/<行业>/<YYYY-MM-DD>_<主题>_<来源简称>.md
```

用 `find` 或 `grep` 检索，不做中央 index（轻量原则）。

### 归档触发规则

**当 WebSearch 结果被写进 wiki 正文时**，同步归档到 `raw/articles/`。具体触发：

| 场景 | 归档 |
|---|---|
| ingest 财报/政策 PDF | ✅ 走 `raw/reports/` 已有流程 |
| periodic-review 引用的新财报/政策 | ✅ |
| value-invest 分析中写进 wiki 的数据 | ✅ |
| red-blue-debate 的每条论据 | ✅（论据必须有归档出处）|
| 日常 query / 股价查询 / 八卦性问答 | ❌ |

### 数据准确性强约束（硬要求）

落库的每条数据**必须**满足：

1. **原文摘录可核对**：关键数字必须以**原文完整句子**的形式引用，加引号。不得基于 WebSearch 的 AI 总结写入——AI 总结可能有失真，必须 WebFetch 原 URL 回访核对
2. **原始 URL 完整**：`source` 字段必须是**具体文章 URL**，不得只写域名或搜索入口 URL
3. **只归档原文披露的数字**：如果原文只提了 A 数据没提 B 数据，不得把 B 混进这个归档（B 另立归档文件）。**禁止在一篇归档里塞进别篇的数据**
4. **抓取日期 ISO 格式**：`fetched: YYYY-MM-DD`，不得用相对日期
5. **状态字段**：`verified: true / partial / unverified`，三档含义：
   - `true` — 已 WebFetch 原 URL 核对，所有数字一字一致
   - `partial` — 部分数字核对过，其余来自 WebSearch 摘要
   - `unverified` — 未做回访核对（临时落库，应尽快转 true）
6. **存在矛盾必须保留两说**：如果同一数据点不同来源不一致（如"15,263 vs 15,251"），**两个版本都归档**，在 frontmatter 的 `conflicts` 字段里互相指向，不得单方面修正

### 文件格式

```markdown
---
source: https://domain.com/path/to/article
fetched: 2026-04-24
title: 原文标题（一字不改）
publisher: 来源（新浪财经 / 上海证券报 / 公司公告等）
published: 2026-04-01（原文发布日，可从页面读取）
stocks: [中信证券]
topics: [Q1财报, 2026Q1]
verified: true
conflicts: []  # 如有，填充冲突文件路径列表
---

# 原文标题

## 关键数据（原文已验证）

- {数据点}：{数字}（来自原文 "{完整句子引用}"）
- ...

## 原文摘录

> {关键段落引号引用}

## 被引用于

- wiki/path/to/page.md#章节锚点
- wiki/path/to/another.md#章节锚点
```

### 反向引用

wiki 页面的"信息来源"或"数据来源"章节，引用时同时写上 `raw/articles/` 路径：
```
- [原文标题](URL) — 已归档 `raw/articles/stocks/中信证券/2026-04-09_Q1业绩快报_证券时报.md`
```

### 不归档什么（避免垃圾场）

- 一次性股价波动、单日新闻
- 纯观点性营销文（"机构看好"）
- 只是搜索结果页的入口链接（无内容实质）
- 被 wiki 消化且不会再回看的老数据（>3 年且已被新数据替代）
- 低信号的重复报道（同一事实的 5 篇不同媒体报道，选信号最强的 1-2 篇归档即可）

## 语言

所有 wiki 页面以**中文**撰写。原始资料可能是英文，提取时翻译为中文。股票代码、专有名词可保留英文。

## 目录结构

```
raw/
  reports/    # 研报、财报 PDF
  articles/   # 网页文章（markdown 格式）
  notes/      # 个人笔记、交易记录
wiki/
  index.md    # 内容目录索引（每个页面一行）
  log.md      # 操作日志
  stocks/     # 个股研究页（每只股票一个文件）
    focus/    # 重点关注个股（符号链接→上级同名文件）
  funds/      # 基金/ETF 研究页
  macro/      # 宏观经济主题页
  sectors/    # 行业/板块分析页
  strategies/ # 投资策略与框架
  journal/    # 投资日志与复盘
```

## 重点关注列表（全局约束）

`wiki/stocks/focus/` 目录存放重点关注个股的**符号链接**，指向 `wiki/stocks/` 下的同名文件。

- **所有需要消费关注列表的 skill**（如 periodic-review、value-invest 等）统一从此目录获取列表，不得各自维护独立的关注列表
- 添加关注：`ln -sf ../xxx.md wiki/stocks/focus/`
- 移除关注：删除 `wiki/stocks/focus/` 中对应的符号链接
- `wiki/stocks/` 中未被链接到 `focus/` 的个股仍保留分析页面，只是不在定期复盘等流程中自动追踪

## 命名规范

- 个股文件：A股用中文名 `贵州茅台.md`，美股用代码 `NVDA.md`
- 基金文件：`{基金简称}.md`，如 `沪深300ETF.md`
- 宏观文件：用主题命名，如 `美联储利率周期.md`
- 行业文件：`{行业名}.md`，如 `半导体.md`
- 策略文件：`{策略名}.md`，如 `股债平衡.md`
- 日志文件：`{YYYY-MM-DD}.md`，如 `2026-04-12.md`

## 页面模板

创建新页面时，读取 `templates/页面模板.md` 获取模板格式（个股、基金、宏观、行业、日志）。

## 工作流

### Ingest（摄入）

当用户提供新资料（PDF、URL、笔记）时：

1. **读取内容**：读取 `raw/` 中的文件或抓取 URL 内容
2. **提取关键信息**：识别涉及的公司、行业、宏观主题、数据点
3. **【财报PDF专属】生成结构化摘要** — 避免重复解析 PDF：
   1. 读取 `templates/财报摘要模板.md` 获取格式
   2. 抽取到 `raw/reports/<公司>_<报告期>.md`（与 PDF 同名），每个数据点标注 PDF 页码（如 `营收1688.38亿(p.6)`）
   3. 摘要末尾**自动生成"核对清单"**（所有量化数据行），每项含：数据点 / 值 / 引用页 / 状态
4. **【财报PDF专属】审查步骤** — 隔断加工错误：
   1. 按核对清单逐项 `Read pages=X` 回读 PDF 对应页
   2. 比对数字/单位/同比/口径，更新状态列（✓ 一致 / ⚠️ 偏差）
   3. 发现偏差必须**修正摘要 md**，不得带偏差继续下游
   4. 审查通过后在 frontmatter 设置 `reviewed: true` + `reviewer_note`
   5. **禁止跳过审查**直接回填 wiki
5. **更新 wiki 页面**（基于已审查的摘要）：
   - 如果相关页面已存在 → 更新对应章节，追加新信息
   - 如果页面不存在 → 按模板创建新页面
6. **维护交叉引用**：确保页面间的链接正确
7. **更新 index.md**：添加新页面条目
8. **更新 log.md**：记录本次摄入操作

注意：
- 不要删除已有信息，追加或修正
- 标注信息来源和日期
- 如果新信息与已有信息矛盾，两者都保留并标注
- 二次消费同一份财报时，**优先读取已审查的 `.md` 摘要**而非重复解析 PDF

### Query（查询）

当用户提出投资问题时：

1. **搜索相关页面**：在 `wiki/` 中搜索关键词
2. **综合回答**：基于多个页面综合答案
3. **引用来源**：标注信息来自哪些 wiki 页面
4. **回写有价值的洞察**：如果综合分析产生了新的洞察，写入相关页面或创建新页面
5. **记录日志**：在 `log.md` 中记录查询

### Lint（检查）

定期维护 wiki 健康度：

1. **时效性检查**：标记超过 3 个月未更新的页面
2. **索引同步**：检查 `index.md` 是否覆盖所有 wiki 文件
3. **链接检查**：验证所有交叉引用链接指向存在的文件
4. **矛盾检测**：标记不同页面间可能矛盾的信息
5. **空白检测**：标记模板字段仍为占位符的页面
6. **输出报告**：汇总所有问题，建议修复方案

## 标签规范

在页面顶部用 YAML frontmatter 标注标签：

```yaml
---
tags: [A股, 白酒, 消费]
updated: 2026-04-12
---
```

## 写作风格

- 简洁精炼，避免废话
- 用数据说话，尽量包含具体数字
- 观点标注来源，区分事实和观点
- 使用 bullet points 提高可扫描性
