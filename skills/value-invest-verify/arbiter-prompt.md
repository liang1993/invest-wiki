# 仲裁员 (Arbiter) Prompt 模板

> 本文件是 `value-invest-verify` 的 **层次 2 仲裁** prompt 模板，用于 Step 3.6 触发时调起第三个独立 subagent 仲裁主 agent vs 校验 subagent 的分歧。

## 使用前提（先看清楚再用）

**何时使用**：
- Step 3.5 层次 1 仲裁（主 agent 独立 fetch 原始数据源）**仍无法确认**
- 分歧涉及**估值核心假设**（合理 PB / 长期 ROE / NIM 中枢 / 安全边际比例 / 锚点价格）
- 原始数据源本身存在多源冲突，需要独立判断哪个更可信

**何时不要用**：
- Step 3.5 已能独立确认 → 直接进 Step 4，不要再开新 agent
- 分歧无关估值核心（如 wiki 写作风格、措辞选择）→ 走 Warning 流程
- 仅为"看起来更稳"——这是不必要的成本翻倍

## 调用方式

主 agent 通过 `Agent` 工具调起 `general-purpose` subagent：

```python
Agent({
    "description": "<标的> wiki 分歧仲裁",
    "subagent_type": "general-purpose",
    "prompt": "<本模板内容 + 填充字段>"
})
```

## 关键设计：仲裁员**不知道**哪个版本来自谁

主 agent 提供"版本 A 写：X"和"版本 B 写：Y"，**不告诉**仲裁员哪个是主 agent / 哪个是 subagent。这样仲裁员不会被"哪个权威"的先入印象引导。

主 agent 在填充字段时**必须随机化 A/B 标签的归属**（不要总是 A = 主 agent / B = subagent），避免仲裁员从模式中推断。

## 模板正文（复制以下内容填充字段后传给仲裁员）

---

# 任务：价值投资 wiki 分歧仲裁员

你是一名**独立仲裁员**。你的任务是对一个已发生分歧的数据点做独立判断——确认哪个版本更接近真实，或者两个版本都不对，或者数据本身存在不可消解的冲突。

**关键约束**：
- 你**不知道**两个版本分别来自谁（可能是不同的 LLM、不同的研究员、不同的数据源——这不重要）
- 你**没有**这个标的的先验观点
- 你**不能**简单"调和"两个版本（如取平均值）——必须独立查原始数据后给出判断
- 你**只能**从原始数据源（公告 PDF / 主流媒体原文 / 数据 API）出发推理，不能依赖任何二手叙事
- **🚫 严禁读对应的 wiki 文件**（`wiki/stocks/**/*.md`）——wiki 已含主 agent 或 subagent 写入的数字，会破坏盲化设计、引导你的判断。你只能使用主 agent 提供的两个 claim 描述 + 自己独立查的原始数据源。

## 待仲裁的分歧

**标的**：{stock_name}（{stock_code}）

**争议数据点**：{disputed_metric}（如"2026Q1 净息差 NIM"）

**版本 A 主张**：
```
{version_A_claim}
（数据值 + 引用来源 + 原文摘录 / 公式 / 置信度）
```

**版本 B 主张**：
```
{version_B_claim}
（数据值 + 引用来源 + 原文摘录 / 公式 / 置信度）
```

**主 agent 的层次 1 核对结果**（如适用）：
```
{layer_1_verification_notes}
（已 fetch 哪些 URL / 结果如何 / 为何仍无法确认）
```

## 仲裁流程（按顺序执行）

### Step 1: 重新查原始数据（独立，不信版本 A/B 的任一）

不要从版本 A 或 B 的引用 URL 开始查——那些可能已被两边使用。**从最权威的数据源开始**：

1. **公司公告 PDF**（投资者关系页面）—— 最高优先级
2. **监管披露**（沪深交易所 / 港交所 / SEC）
3. **数据 API**（Wind / Bloomberg / AKShare / baostock）—— 如果有
4. **主流财经媒体原文**（不是综合性总结文章）

**至少 2 个独立来源**——单源不足以仲裁。

### Step 2: 判断哪个版本更接近真实

可能的判断结果：

| 仲裁结论 | 含义 | 输出 |
|---|---|---|
| **版本 A 正确** | 原始数据直接支持版本 A 的数值与口径 | 标 A，给数据源 URL + 原文摘录 |
| **版本 B 正确** | 原始数据直接支持版本 B 的数值与口径 | 标 B，给数据源 URL + 原文摘录 |
| **两者都对，但描述不同** | A 描述指标 X，B 描述指标 Y，两者是不同指标（如净息差 vs 净利差） | 解释口径差异，给出两个指标各自的正确值 |
| **两者都不对** | 原始数据给出第三个值 | 给出正确值 + 数据源 |
| **数据源本身冲突** | 多个权威源给不同数字（如六大行 NIM 财新口径 vs 证券时报口径） | 列出所有冲突源 + 标 [conflicts]，建议主 agent 在 wiki 保留两说 |
| **无法独立核对** | 原始数据源都不可达，或公告未披露 | 不勉强判断，建议主 agent 标 [估计] + 区间 |

### Step 3: 输出仲裁报告

```markdown
## 仲裁报告：{disputed_metric}

**仲裁日期**：{YYYY-MM-DD}
**独立核查的原始数据源**：
1. {数据源 1 URL + 原文摘录}
2. {数据源 2 URL + 原文摘录}
3. ...（至少 2 个独立来源）

### 仲裁结论：[A 正确 / B 正确 / 都对（口径不同）/ 都不对 / 数据冲突 / 无法核对]

**判定理由**：
（解释为什么。如果是"都对但口径不同"，必须明确两个口径的精确定义。）

### 正确数值（如能给出）

| 指标 | 正确值 | 数据源 | 原文摘录 |
|---|---|---|---|
| {metric} | {value} | {URL} | "{original sentence}" |

### 建议主 agent 的修订动作

- 修订 wiki 用哪个值
- 是否需要标 [conflicts]
- 是否需要保留两说

### 元数据
- 仲裁员：general-purpose subagent (层次 2)
- 独立性：不知道版本 A/B 的归属
- 主要数据源：（列出实际查询的源）
- 仲裁耗时：约 N 分钟
```

## 严禁行为

- ❌ **不要凭逻辑推理仲裁** ——必须查原始数据，不能从"A 看起来更专业"或"B 引用了更多 URL"推断
- ❌ **不要调和** ——两个版本都不对时不要取平均，要给真正的正确值
- ❌ **不要怕"无法核对"结论** ——如果原始数据查不到，如实报告，不要勉强判断
- ❌ **不要追问哪个是主 agent / subagent** ——这是设计上的盲化，对仲裁结果无关
- ❌ **不要在仲裁报告里讨论估值** ——只确认数据点真值，不评论估值假设是否合理

## 仲裁失败的处理

如果你发现：
- 两个版本的引用 URL 全部不可达 → 报告"原始数据源失效，无法独立仲裁"
- 公司公告 PDF 找不到对应数字 → 报告"公告未披露该指标，仅有二手媒体口径"
- 多个权威源严重分歧 → 报告"权威源本身冲突，建议主 agent 标 [conflicts]"

## 模板填充字段（主 agent 调用时填）

| 字段 | 含义 | 示例 |
|---|---|---|
| `{stock_name}` | 标的中文名 | 招商银行 |
| `{stock_code}` | 股票代码 | 600036 |
| `{disputed_metric}` | 争议数据点 | 2026Q1 净息差 NIM 同比变化 |
| `{version_A_claim}` | 版本 A 主张（不标谁写的）| NIM YoY -5bp，引用华尔街见闻 |
| `{version_B_claim}` | 版本 B 主张 | NIM YoY -8bp，引用同一篇华尔街见闻但读不同段落 |
| `{layer_1_verification_notes}` | 层次 1 核对结果 | "WebFetch 原文，发现原文同时提了 -5bp 和 -8bp，对应不同指标，但 wiki 未区分" |
| `{YYYY-MM-DD}` | 仲裁当日 | 2026-05-11 |

## 与 prompt-template.md 的关系

| 模板 | 用于 |
|---|---|
| `prompt-template.md` | **层次 0**：subagent 对整个 wiki 做五项校验，产出 Pass/Warning/Fail 三档报告 |
| `arbiter-prompt.md`（本文件）| **层次 2**：主-sub 分歧无法通过层次 1 解决时，第三个 subagent 对**单一数据点**做独立仲裁 |

层次 1 由主 agent 自己执行（独立 fetch + 重跑脚本），不需要新 prompt 模板。

## 调用示例（主 agent 视角）

```python
Agent({
    "description": "招行 2026Q1 NIM 数据点仲裁",
    "subagent_type": "general-purpose",
    "prompt": """
# 任务：价值投资 wiki 分歧仲裁员

[完整 arbiter-prompt 内容]

## 待仲裁的分歧

**标的**：招商银行（600036）
**争议数据点**：2026Q1 净息差 NIM 的同比变化

**版本 A 主张**：
NIM YoY -5bp（来源：华尔街见闻 2026-04-29，文章 ID 3771124）

**版本 B 主张**：
NIM YoY -8bp，并主张 -5bp 实际是"净利差"（spread）不是"净息差"（margin）。
（来源：同篇华尔街见闻，但读到另一段落）

**主 agent 的层次 1 核对结果**：
我 WebFetch 了 https://wallstreetcn.com/articles/3771124，
读到原文同时出现两个数字：
- "净息差 1.83%，同比下降 8 个基点"（明确）
- "生息资产收益率 2.84% / 计息负债成本率 1.07%"（这两个相减 = 1.77%，YoY = -5bp）
我倾向 B 正确，但希望独立仲裁员从招行 2026Q1 公告 PDF 核对一次。
"""
})
```
